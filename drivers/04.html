MODULE 04

Kernel Timer Driver

Async event generation

timer_listsoftirq contextspinlock_irqsavedel_timer_sync

Overview

A kernel timer fires every 1000ms and writes timer_event_N into a ring buffer. Userspace reads events via blocking read(). Timer start/stop is controlled via ioctl(). Critical constraint: timer callbacks run in softirq context — mutex is forbidden, spinlock is mandatory.

ParameterValue
Device node/dev/sanath_timer
Timer interval1000 ms
Synchronizationspinlock_t with irqsave/irqrestore
Event formattimer_event_N

Spinlock vs Mutex — Context Matters

PrimitiveCan SleepSafe in Softirq
mutexYes❌ No
spinlock_tNo✅ Yes
unsigned long flags;
spin_lock_irqsave(&kernel_logger.lock, flags);
/* critical section */
spin_unlock_irqrestore(&kernel_logger.lock, flags);
irqsave/irqrestore saves the CPU interrupt enable state before locking and restores it exactly on unlock.

What Cannot Run Under Spinlock

FunctionReasonSafe?
wake_up_interruptibleTouches scheduler❌ Move outside
copy_to_userMay page fault❌ Never
kmalloc(GFP_KERNEL)May sleep❌ Never
mod_timerHas internal locks⚠️ Move outside
snprintf / memcpyNo side effects✅ Yes

Correct copy_to_user pattern

uint8_t tmp[MEM_SIZE];
spin_lock_irqsave(&lock, flags);
memcpy(tmp, buffer[idx], copy_len);
count--;
spin_unlock_irqrestore(&lock, flags);
if (copy_to_user(user_buf, tmp, copy_len))
    return -EFAULT;

Bugs Encountered and Fixed

BugFix
copy_to_user under spinlockMoved outside — use local tmp[] buffer
wake_up under spinlockMoved outside critical section
Timer not stoppingAdded timer_active flag checked in callback
del_timer on exitReplaced with del_timer_sync
Using del_timer instead of del_timer_sync is a kernel crash waiting to happen — the callback may still be executing on another CPU when the module unloads.
